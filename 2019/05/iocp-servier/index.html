基于IOCP技术的网络服务器

May 25, 2019 in windows, net

网络通信模型中性能比较好的模型linux下有epoll, windows下则有IOCP. 前段时间一直想学习一下windows下的IOCP模型。在此记录一下学习过程中的各种问题。

一. 为什么要使用IOCP

  iocp是windows下性能最高的网络通信模型, 传统通信模型中(同步)通常对一个网络连接的读写都是在一个线程中实现,在此期间线程不能做其它事情,只能阻塞等待对网络的读写操作完成后才能做其它事情,如果有大量连接入则需要启动大量的线程。 相比之下,IOCP只需要几个线程来处理网络操作, 网络通信过程中的读写完全实现了异步，不需要等待。最大限度的减少了线程间的上下文切换.且IOCP在网络数据接收到后会先放入我们事先准备好的缓冲区中，然后才通知工作线程有新数据到达,工作线程完全不需要重新再拷贝一次数据,最大限度的减少耗时。

二. 使用IOCP

  使用IOCP一般是先在主线程中创建一个IOCP,然后根据CPU的核心数.创建对应数量的工作线程(CPU核心数*2).后续过程中将socket不断的绑定到这个IOCP中.下图是IOCP的整个工作流程.
在这个流程中有几个关键点需要讲一下：
1. post accept这里要求异步,因为我们只是告诉iocp我已经准备接收新的连接了,有新的连接到来时通知我,我好做相应处理.在这过程中我不要阻塞还可以做其它事情。那么怎么实现异步accept呢?使用windows提供的AcceptEx函数,此函数在windows系统提供的扩展函数.这里推荐先使用WSAIoctl先获取到AcceptEx函数的指针.后续直接使用,获取AcceptEx的代码：

GUID GuidAcceptEx = WSAID_ACCEPTEX;
GUID GuidGetAcceptExSockAddrs = WSAID_GETACCEPTEXSOCKADDRS;
WSAIoctl(m_ListenSock, SIO_GET_EXTENSION_FUNCTION_POINTER, &GuidAcceptEx,
sizeof(GuidAcceptEx),	&m_lpfnAcceptEx, sizeof(m_lpfnAcceptEx), &dwBytes, NULL, NULL);

第一个socket需要一个有效的socket即可,在使用AcceptEx时,需要先准备一个socket,原因是：异步调用，当收到通知时，直接使用这个socket,不需要在当场重新建立一个新的socket(创建一个新的socket是相当耗资源的), AcceptEx参数如下：

BOOL AcceptEx ( 	
                  SOCKET sListenSocket, 
                  SOCKET sAcceptSocket, 
                  PVOID lpOutputBuffer, 
                  DWORD dwReceiveDataLength, 
                  DWORD dwLocalAddressLength, 
                  DWORD dwRemoteAddressLength, 
                  LPDWORD lpdwBytesReceived, 
                  LPOVERLAPPED lpOverlapped 
   );

这里主要说明一下AcceptEx默认是连接到达后,第一笔数据到达才返回通知应该程序,如果希望新连接到达就通知应用程序而不需要等待第一笔数据到达，则需要将dwReceiveDataLength参数填为0.表示不需要同时接收第一笔数据.最后一个参数lpOverlapped是IOCP特有的重叠操作。这个重叠结构是实现异步操作核心，可以理解为这个结构标识了当前IO操作是属于哪个请求的。还有一个函数GetAcceptExSockAddrs也需要同AcceptEx一样操作取对应的函数指针,这函数的作用就是,当有新连接到来时,应该用程序通过此函数来获取新连接。取函数指针的代码：

WSAIoctl(m_ListenSock, SIO_GET_EXTENSION_FUNCTION_POINTER, &GuidGetAcceptExSockAddrs,sizeof(GuidGetAcceptExSockAddrs), &m_pGetAccepetAddrsFunc, sizeof(m_pGetAccepetAddrsFunc), &dwBytes, NULL, NULL)

这个函数GetAcceptExSockAddrs就专门就来解析AcceptEx的能同时解析新连接第一次的数据,如果需要连接到达就通知而不需要等待数据,则需要将参数第二个参数置为0.新连接到达时，调用此函数的代码为：

sockaddr_in* pClientAddr = NULL;
sockaddr_in* pLocalAddr = NULL;
DWORD dwBytes = 0;
int remoteLen = sizeof(sockaddr_in), localLen = sizeof(sockaddr_in);
m_pGetAccepetAddrsFunc(pIoConText->m_wsaBuf.buf, 0,sizeof(SOCKADDR_IN) + 16, sizeof(SOCKADDR_IN) + 16,	(LPSOCKADDR*)&pLocalAddr,&localLen, (LPSOCKADDR*)&pClientAddr, &remoteLen);

pClientAddr是客户端信息、pLocalAddr为本机IP信息,pIoConText->m_wsaBuf.buf就是用来储存第一次传入的数据的,如果第二个参数置0后,则此缓冲区中就不会有数据。

三. 具体过程

  1. 创建IOCP
    创建IOCP是比较简单的,代码如下:

    m_hCompletePort = CreateIoCompletionPort(INVALID_HANDLE_VALUE, NULL, 0, 0);

    直接调用CreateIoCompletionPort即可,调用成功返回的就是IOCP句柄.后期其它socket在绑定时就使用这个句柄.

  2. 将listen socket绑定到IOCP

    // 将用于和客户端通信的SOCKET绑定到完成端口中
HANDLE hTemp = CreateIoCompletionPort((HANDLE)pBaseEvent->GetSocket(), m_hCompletePort, (DWORD)pBaseEvent, 0);

    注意一下CreateIoCompletionPort函数中的第三个参数pBaseEvent,这是一个对象指针.后期当这个socket有触发对应的事件时,这个指针会返回.那么我们就可以通过这个指针来回调处理对应的事件啦.

  3. 创建工作线程
    这里就直接使用C++11的线程类来实现：

    for (int i = 0; i < m_iThreadCount; i++)
{
m_pThreadArray[i] = std::thread(&NetCore::WorkThread, this);
}

    在创建完成工作线程后,就要开始post Accept的请求啦,告诉IOCP我准备好了,可以开始接受请求啦.代码如下：

    DWORD dwBytes = 0;
pPerIoContext = GetNewPerIoContext();
pPerIoContext->m_sockAccept = WSASocket(AF_INET, SOCK_STREAM, IPPROTO_TCP, NULL, 0, WSA_FLAG_OVERLAPPED); 
if (pPerIoContext->m_sockAccept <= 0)
{
m_NetPublic.GetLastErrorInfo(m_sErrorInfo, sizeof(m_sErrorInfo));
Log(ERROR, "post accept socket fail:%s", m_sErrorInfo);
return -1;
}
pIoConText->ReSetBuffer();
pIoConText->m_OpType = ACCEPT_POSTED;
/*AcceptEx 与 GETACCEPTEXSOCKADDRS 是关联起来用的 如果想接收到客户的接入信息(没有发送数据)
则需要将buff长度填为0 表示只接收连接消息
*/
if (FALSE == m_lpfnAcceptEx(m_ListenSock, pIoConText->m_sockAccept, pIoConText->m_wsaBuf.buf,
0,  sizeof(SOCKADDR_IN) + 16, sizeof(SOCKADDR_IN) + 16, &dwBytes, &pIoConText->m_Overlapped))
{
if (WSA_IO_PENDING != WSAGetLastError())
{
m_NetPublic.GetLastErrorInfo(m_sErrorInfo, sizeof(m_sErrorInfo));
Log(ERROR, "post accept.错误原因:%s", m_sErrorInfo);
return -5;
}
}

    这里需要解释一下,前面说过在调用AcceptEx时要事先准备好socket,有连接到来就系统就可以直接用了,在创建socket时要使用WSASocket函数,最重要的是第四个参数WSA_FLAG_OVERLAPPED,这是重叠IO必须要使用的.PER_IO_CONTEXT是IO请求结构体,里面包含重叠IO结构体,其它字段根据自己需要定义：

    class PER_IO_CONTEXT
{
public:
	PER_IO_CONTEXT();
	void ReSetBuffer();
	virtual ~PER_IO_CONTEXT();
public:
	OVERLAPPED     m_Overlapped;                               // 每一个重叠网络操作的重叠结构(针对每一个Socket的每一个操作，都要有一个)        
	SOCKET         m_sockAccept;                               // 这个网络操作所使用的Socket
	WSABUF         m_wsaBuf;                                   // WSA类型的缓冲区，用于给重叠操作传参数的
	char           m_szBuffer[MAX_BUFFER_LEN];                 // 这个是WSABUF里具体存字符的缓冲区
	OPERATION_TYPE m_OpType;                                   // 标识网络操作的类型(对应上面的枚举)

};

 到这里主线程的事情基本算是完了.下面看看工作线程做了些什么
4. 工作线程
工作线程启动后就进入一个循环中,调用GetQueuedCompletionStatus等待所有注册到IOCP中的socket产生的任何事件.

  while (WAIT_OBJECT_0 != WaitForSingleObject(m_hShutdownEvent, 0))
  {
    bReturn = GetQueuedCompletionStatus(m_hCompletePort, &dwBytesTransfered, (PULONG_PTR)&pBaseEvent, &pOverlapped, INFINITE);
    // 如果收到的是退出标志，则直接退出
    if (EXIT_CODE == (DWORD)pBaseEvent)
    {
      break;
    }
    if (!bReturn)
    {
      m_NetPublic.GetLastErrorInfo(m_sErrorInfo, sizeof(m_sErrorInfo));
      if (NULL != m_pCloseConnect)
      {
        iRet = m_pCloseConnect(pBaseEvent->GetOnlyName());
      }
    }
    else
    {
      // 判断是否有客户端断开了
      PER_IO_CONTEXT* pIoContext = CONTAINING_RECORD(pOverlapped, PER_IO_CONTEXT, m_Overlapped);
      if ((0 == dwBytesTransfered) && (RECV_POSTED == pIoContext->m_OpType || SEND_POSTED == pIoContext->m_OpType))
      {
        iRet = m_pCloseConnect(pBaseEvent->GetOnlyName());
      }
      else
      {
        pBaseEvent->DispatchEvent(pIoContext);
      }
    }
  }

虽然有多个线程在调用这个函数,但IOCP保证同一个socket只会有一个线程会调用.我们可以根据请求IO结构体中的操作类型来判断当前是什么请求(新连接接入/数据到达/写数据完成),然后通过我们在绑定IOCP时传入的事件类指针回调到对应的事件处理函数。这里有一个点需要注意，就是在对应的处理事件中,在数据处理完成后,要重新post对就的请求,表明准备好接收下一个请求了.
5. 工作线程退出
在工作线程中,循环标志就是一个信号量WaitForSingleObject用于判断是否继续处理网络请求，当服务器退出时，调用： SetEvent这样循环就不会继续下去了。 但这样会有问题,当线程阻塞在GetQueuedCompletionStatus函数时,调用SetEvent并不会让线程马上结束,解决办法是调用PostQueuedCompletionStatus函数，这个函数可用给IOCP post一个请求,至于请求内容是啥并不限定.函数原型如下：

  BOOL WINAPI PostQueuedCompletionStatus(
                      __in      HANDLE CompletionPort,
                      __in      DWORD dwNumberOfBytesTransferred,
                      __in      ULONG_PTR dwCompletionKey,
                      __in_opt  LPOVERLAPPED lpOverlapped
   );

这里注意一下第三个参数,我们在调用PostQueuedCompletionStatus我们可以传入NULL,或者传入自己约定的一个内容,这样工作线程中解析到这个值就知道要退出了.由于调用一次只会唤醒一个工作线程,所以程序启动了多少个工作线程就需要调用多少次。

  if (!m_bIsEnd)
  {
    for (int i = 0; i < m_iThreadCount; i++)
    {
      PostQueuedCompletionStatus(m_hCompletePort, 0, (DWORD)EXIT_CODE, NULL);
    }
  }
  return 0;

这样就可以停掉工作线程了。
6. 心跳
服务端肯定是需要心跳的,最开始时采用的是tcp带外数据的方式发送心跳,但在连续发送17次时就会被windows给断掉.并且是windows7以后才有的问题。最后采用的是发送0字节的数据到客户端,判断send的返回值,如果小于0,则说明连接有问题,重试N次,如果都不能发送成功,则将此连接断掉.

四. 总结

IOCP在异步接收连接请求时，与普通方式最大的区别在于IOCP需要等第一笔数据到达后才通知程序,这样在某些场景下可能有问题,最终还是通过参数控制,使用IOCP在接收到新连接就立即返回,后续交给服务处理.还有一个点是,客户端发送数据后,服务接收到recv通知时,实际这笔数据IOCP已经给你保存到你事先准备的缓冲区中了.不需要再拷贝一次.
